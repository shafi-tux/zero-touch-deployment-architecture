CI/CD AUTOMATION

Deployment Pipeline

Scroll to trace the fully automated CI/CD workflow — from git push to live production in under 60 seconds

01

Source Control Trigger

A git push to the main branch automatically triggers the GitHub Actions workflow — no manual intervention required.

Trigger on: push → branches: [main]
02

Repository Checkout

The CI runner provisions a clean Ubuntu environment and clones the repository via actions/checkout@v6.

Action actions/checkout@v6
03

Secure Remote Access

The pipeline authenticates to the production EC2 instance via SSH using a private key securely stored in GitHub Secrets.

Secrets EC2_HOST, EC2_SSH_KEY, REDIS_HOST
04

Runtime Configuration

The private IP of the database server is dynamically injected as a .env variable, ensuring the application connects to the correct backend at deploy time.

Command echo "REDIS_HOST=" > .env
05

Container Orchestration

Docker Compose builds fresh images and orchestrates the Nginx reverse proxy and Python application containers with zero-downtime deployment.

Command docker compose up --detach --build
06

Production Traffic Serving

The application is live and serving HTTP traffic through the Nginx reverse proxy on port 80 — fully operational, zero manual steps.

Status HTTP 200 — Production Ready

LEAST-PRIVILEGE MODEL

Security Architecture

Defense-in-depth strategy implementing network segmentation, security group chaining, and least-privilege access control

Layer 1

Edge — Internet Gateway

Only HTTP (port 80) and SSH (port 22) traffic is allowed inbound. All other ports are implicitly denied by AWS default rules.

IN TCP :80 — HTTP 0.0.0.0/0
IN TCP :22 — SSH 0.0.0.0/0
IN All Other — DENIED
Layer 2

Network — Subnet Isolation

Database server sits in a private subnet with no public IP and no inbound internet route. Only the NAT gateway provides outbound access for system updates.

OUT Private → NAT → Internet Updates only
IN Internet → Private — BLOCKED
Layer 3

Application — Security Group Chaining

The database Security Group only accepts connections from the web server's specific Security Group ID — not from any IP range.

IN TCP :6379 — Redis sg-webserver
IN TCP :22 — SSH sg-webserver
IN All Other Sources — DENIED
Layer 4

Transport — Reverse Proxy

Nginx acts as a gateway shielding the Python application from direct internet exposure. The app container only exposes port 8000 internally — never to the host.

→ Client → Nginx :80 → Python :8000
✕ Client → Python :8000 — NOT EXPOSED
